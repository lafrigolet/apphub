# ADR 003 — Dynamic NGINX routing via Redis sidecar

**Date**: 2026-04-28
**Status**: Accepted
**Supersedes (partially)**: the static `infra/nginx/conf.d/<subdomain>.conf` pattern described in ARCHITECTURE.md

## Context

When staff registers a new app from voragine-console (`POST /v1/apps`), the platform needs to
serve traffic on the new subdomain immediately. The original setup required:

1. Hand-writing `infra/nginx/conf.d/<subdomain>.conf` on the host
2. `docker compose restart nginx` (or `nginx -s reload` inside the container)

That breaks the "self-service" model — staff cannot complete the app provisioning end-to-end
from the UI. It also does not survive multi-node deployments: a `conf.d/` bind mount lives on
one host's filesystem; replicas of NGINX on other nodes would not see the new file. `inotify`
watchers on shared filesystems (NFS, CephFS) are unreliable.

We needed a model that:

- Lets `platform-core` add/remove NGINX server blocks programmatically with no host-side ops
- Propagates changes to **every** NGINX replica in a cluster within seconds
- Tolerates nodes joining/leaving without manual sync
- Survives short Redis outages without dropping existing traffic
- Keeps NGINX vanilla (no OpenResty, no NGINX Plus, no proprietary admin API)

## Decision

Use **Redis as the source of truth for per-subdomain server blocks** and run a polling
sidecar inside the same NGINX container. The sidecar reconciles a Redis hash with the local
`/etc/nginx/conf.d/sites/` directory and triggers `nginx -s reload` whenever the contents
change.

```
┌──────────────── NGINX container (one per node) ────────────────┐
│                                                                │
│   nginx ──reads── /etc/nginx/conf.d/sites/*.conf               │
│      ▲                ▲                                        │
│      │ SIGHUP         │ writes                                 │
│      │                │                                        │
│   sidecar.sh ─poll 2s─┘                                        │
│      │                                                         │
└──────┼─────────────────────────────────────────────────────────┘
       │ HGETALL nginx:configs
       ▼
   ┌─────────┐  HSET nginx:configs <subdomain> <conf>
   │  Redis  │ ◄─────── platform-core (on POST /v1/apps)
   └─────────┘
```

### Components

| Component | Responsibility |
|---|---|
| `infra/nginx/Dockerfile` | Custom image: `nginx:alpine` + `redis-cli` + `tini`. Bakes seed configs and the sidecar scripts |
| `infra/nginx/seed/*.conf` | Static seed configs for the platform's own portals (yoga, splitpay, aikikan, voragine-console, portal). Used to populate Redis on first boot of an empty cluster |
| `infra/nginx/entrypoint.sh` | PID-1 entrypoint: runs `sidecar.sh init` synchronously, then `sidecar.sh watch &`, then `exec nginx -g 'daemon off;'` |
| `infra/nginx/sidecar.sh` | `init` waits for Redis, seeds the hash if empty, renders to disk. `watch` polls every `POLL_INTERVAL` seconds, hashes `HGETALL`, re-renders + reloads on change |
| `infra/nginx/nginx.conf` | `include /etc/nginx/conf.d/sites/*.conf` glob — picks up whatever the sidecar wrote |
| `platform/tenant-config/src/services/nginx-config.service.js` | `writeAppNginxConfig({appId, subdomain})` renders the template and does `HSET nginx:configs <subdomain> <conf>` + `PUBLISH nginx:reload <subdomain>` |
| `platform/tenant-config/src/services/apps.service.js` | Calls `writeAppNginxConfig` after a successful `INSERT INTO platform_tenants.apps` (best-effort; failure is logged but does not roll back the DB transaction) |

### Redis schema

```
HKEY  nginx:configs                     → hash, one field per subdomain
HGET  nginx:configs autoroute           → "server { listen 80; server_name autoroute.hulkstein.local …"

PUBSUB nginx:reload                     → channel, payload = subdomain (advisory; sidecar polls anyway)
```

### Convention for portal containers

Every server block generated by `writeAppNginxConfig` proxies `/` to
`<subdomain>-portal:5180`. To make a new app's frontend work, the operator must add a Docker
service named `<subdomain>-portal` listening on 5180. Until then, the subdomain returns
`HTTP 502` for `/` while `/api/*` (platform routes) keep working because they go to
`platform_core`.

The dynamic resolver (`resolver 127.0.0.11 valid=10s`) inside the rendered config means NGINX
does not fail to start when `<subdomain>-portal` does not yet exist; it only returns 502 on
request and recovers as soon as the container appears.

## Rationale

We considered five alternatives:

1. **Filesystem + `inotify`**: works on single host with bind mounts. Breaks on multi-node:
   bind mounts are per-host, `inotify` does not propagate between nodes, NFS introduces
   another SPOF and notification quirks.

2. **Docker Swarm Configs**: native Swarm primitive, replicates via the manager raft store.
   Configs are immutable; updating one forces a rolling redeploy of every NGINX replica
   (recreates containers, drops long-lived connections). Couples `platform-core` to the
   Swarm API socket — wide blast radius.

3. **OpenResty + Lua**: NGINX consults Redis on every request via Lua. Truly dynamic, no
   reload at all. But locks us into OpenResty (different image, different debugging, Lua
   skill required) and replaces declarative server blocks with imperative scripts.

4. **NGINX Plus admin API**: `PATCH /api/8/...` to mutate upstreams in place. Excellent
   technically. Commercial license required.

5. **Kubernetes Ingress Controller**: the canonical answer in K8s. Out of scope while the
   project still ships as plain `docker-compose`.

**Redis + polling sidecar** wins for us because:

- Vanilla NGINX, vanilla Redis — both already in the stack
- Reload (not recreate): preserves existing connections; `nginx -s reload` is graceful
- Cluster-friendly out of the box: every replica polls the same Redis, no inter-node coordination
- Tolerant: Redis outage doesn't drop traffic; new replicas hydrate on arrival
- Trivial to migrate later: when we move to Kubernetes, we replace the sidecar with an
  Ingress controller and `nginx-config.service.js` writes Ingress objects. Everything else
  (UI, DB model, `apps.service.js`) stays.

The trade-off accepted: ~2 second propagation latency. Not noticeable for app provisioning,
which is a human-paced operation.

## Consequences

### Positive

- App registration is a single API call. NGINX self-configures within ~2s. No host shells, no manual reloads.
- Every NGINX replica converges independently. Adding nodes is "scale up the deployment" — no orchestration.
- Auditable: `redis-cli HGETALL nginx:configs` shows exact routing state of the cluster.
- Editable: `redis-cli HSET nginx:configs <subdomain> "$(cat new.conf)"` propagates in 2s.
- Reversible: `HDEL` removes both routing and the rendered file (the sidecar prunes orphans).
- Resilient: invalid configs fail `nginx -t` and the reload is skipped — last good config keeps serving.

### Negative

- Redis becomes a **production dependency for routing changes** (not for serving traffic — running NGINX has its config cached on disk). If Redis is down for an extended period, new apps cannot be registered.
- Polling produces a small constant load: ~1 PING + 1 HGETALL every `POLL_INTERVAL` seconds per NGINX replica. Negligible for any realistic count of apps and replicas.
- Direct edits to `/etc/nginx/conf.d/sites/<file>.conf` inside a running container are reverted on the next sidecar tick. Operators must edit in Redis, not in the filesystem.
- Adds a layer of indirection: troubleshooting a missing route requires checking Redis (`HKEYS`) before looking at NGINX itself.

## Operational notes

### Bootstrap (first cluster boot, Redis empty)

The sidecar sees `HLEN nginx:configs == 0` and seeds the hash from the baked-in
`/etc/nginx/seed/*.conf` files. Subsequent boots find the hash populated and skip seeding.

To force re-seeding: `redis-cli DEL nginx:configs` then restart any NGINX replica.

### Tunables

| Env var | Default | Effect |
|---|---|---|
| `REDIS_HOST` | `redis` | Redis hostname (Docker service name) |
| `REDIS_PORT` | `6379` | Redis port |
| `POLL_INTERVAL` | `2` | Seconds between reconciliation passes |
| `INIT_WAIT_SECS` | `30` | How long the sidecar's `init` waits for Redis before falling back to seed files only |

Lower `POLL_INTERVAL` for faster propagation (cost: more Redis traffic). Higher for less load.

### Debugging

```bash
# What's in Redis right now?
docker compose exec redis redis-cli HKEYS nginx:configs

# Show one subdomain's rendered conf
docker compose exec redis redis-cli HGET nginx:configs autoroute

# Watch the sidecar's reconciliation log
docker compose logs -f nginx | grep sidecar

# Force a re-render manually (no behavioural change, just prints the loop output)
docker compose exec redis redis-cli HSET nginx:configs <subdomain> "$(redis-cli HGET nginx:configs <subdomain>)"
```

### Cluster topology

Single Redis instance is fine for development. Production multi-node should use a Redis HA
setup (Sentinel or Cluster). The sidecar's poll-with-retry handles transient Redis
unavailability; on permanent outage, NGINX continues serving the last successfully-rendered
configuration but new apps cannot be registered until Redis recovers.

### Adding new fields to the template

The template lives in `platform/tenant-config/src/services/nginx-config.service.js` as a
JS string. Changes to it apply only to **future** apps — existing entries in the Redis hash
are unchanged. To re-render existing apps with a new template, run a one-off script that
iterates `HKEYS nginx:configs`, re-renders, and writes back via `HSET`.

## Migration path to Kubernetes

When the project moves to Kubernetes, the swap is localised:

1. Replace `nginx-config.service.js` with an implementation that creates `Ingress` (or
   `HTTPRoute`) resources via the Kubernetes API.
2. Replace the custom NGINX container with an Ingress controller (NGINX-ingress, Traefik, etc.).
3. Delete `sidecar.sh`, `entrypoint.sh`, `seed/`. The Ingress controller handles propagation
   to all replicas via the Kubernetes watch stream.

`apps.service.js`, the UI, the DB schema, and all calling conventions stay unchanged.
